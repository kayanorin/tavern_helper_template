# 小剧场提示词审查报告

> 审查范围：`小剧场格式.txt` + 6 个 `小剧场_*.txt` 文件
> 对照对象：6 个对应的前端渲染脚本 (`脚本/小剧场渲染/*.js`)
> 审查标准：简洁性、准确性、是否误导 AI

---

## 一、🔴 严重：字段名与渲染脚本不匹配

以下问题会直接导致前端渲染时**丢失数据**——提示词告诉 AI 用 A 字段名输出，但脚本用 B 字段名去读。

### 1.1 美食点评 `小剧场_美食点评.txt` ↔ `美食点评.js`

| 提示词字段 | 脚本实际读取的字段    | 后果                                   |
| ---------- | --------------------- | -------------------------------------- |
| `店名`   | `名称` (L325, L338) | Tab 和面板标题显示为「餐厅1」兜底值    |
| `营业`   | `营业时间` (L343)   | 营业信息不显示                         |
| _(无)_   | `简介` (L339)       | 脚本支持但提示词未提供，浪费渲染位     |
| _(无)_   | `排名` (L347)       | 用于决定配色方案，缺失后固定为一种色调 |

**建议修改提示词：**

```diff
午浓时 - 暮醇时 | 夜甘时前半段<restaurant>
-店名：{{餐厅名称}}
+名称：{{餐厅名称}}
+简介：{{一句话概括,可选}}
 菜系：{{菜系类型,如"蒸菜 · 创意料理"}}
 评分：{{1.0-5.0,如4.8}}
 价位：{{¥ / ¥¥ / ¥¥¥}}
 标签：{{逗号分隔,如"GMA二星, 预约制, 诚实之味认证"}}
 地址：{{世界中的地址}}
-营业：{{营业时间,用世界历法}}
+营业时间：{{营业时间,用世界历法，如 午浓时 - 暮醇时 | 夜甘时前半段}}
 招牌菜：{{菜名}}|{{一句话描述,生动有画面感}}
</restaurant>
```

Example 部分也需要同步修改：`店名` → `名称`，`营业` → `营业时间`。

### 1.2 民间传说 `小剧场_民间传说.txt` ↔ `民间传说.js`

| 提示词字段 | 脚本实际读取的字段 | 说明                                                           |
| ---------- | ------------------ | -------------------------------------------------------------- |
| _(无)_   | `标题` (L154)    | 脚本支持 `标题` 字段用于卡片顶部大标题，但提示词未定义此字段 |

**说明：** 脚本 L154 读 `data['标题']`，如果没有则 fallback 到 `data['类型']`。目前提示词的格式里没有 `标题` 字段，传说类内容只会显示"地方传说"或"童谣"这种类型名作为标题，效果不佳。

**建议添加字段：**

```diff
<folklore>
 类型：{{地方传说 / 童谣 / 民间歌谣}}
+标题：{{传说或童谣的名称,如"五舌蛇"、"味阶数数歌"}}
 地区：{{发源地,如 炽炎群岛 / 金麦城 / 霜降高原}}
 流传度：{{广为人知 / 仅在当地流传 / 近乎失传}}
```

## 二、🟡 中度：无效/冗余字段浪费 token

### 2.1 烹饪游戏 `小剧场_烹饪游戏.txt` — `场景` 字段不被解析

脚本 `parseCookingData()` (L47-81) 的 `switch` 语句只处理 `模式`、`食材`、`烹饪方法`、`菜品` 四个 key。提示词中定义的 `场景` 字段被 AI 输出后，完全不会被渲染或使用。

**建议二选一：**

- **A. 删除字段**：从提示词中移除 `场景`，节省 token
- **B. 脚本适配**：在 `parseCookingData()` 中添加 `case '场景'` 并在 UI 中展示

如果选 A：

```diff
<cooking>
 模式：{{普通 / 黑暗 / 竞技}}
-场景：{{当前烹饪场景描述,1句,如 "金麦城集市临时厨台"}}
 食材：...
```

### 2.2 烹饪游戏 — `竞技` 模式定义但不实现

提示词 L23-26 定义了三种模式：`普通`、`黑暗`、`竞技`。但脚本 L66 只判断 `includes('黑暗')`，其他一律按 `普通` 处理。`竞技` 模式不会产生任何差异化表现。

**建议：** 要么从提示词中移除 `竞技` 选项（避免 AI 凭空发挥一套不存在的限时规则），要么在脚本中实现竞技模式的差异化 UI。短期内建议直接删除：

```diff
-模式：{{普通 / 黑暗 / 竞技}}
+模式：{{普通 / 黑暗}}
```

并删除 L23-26 的竞技模式说明。

### 2.3 学术论文 — 多个脚本支持但提示词未定义的字段

脚本 `buildAcademicHTML()` 读取以下字段，但提示词的 Output format 中都没有定义：

| 脚本读取的字段                | 说明                                                     |
| ----------------------------- | -------------------------------------------------------- |
| `单位` (L284)               | 作者所属学院，但提示词把学院写在 `作者` 字段的逗号后面 |
| `职位` (L285)               | 同上                                                     |
| `投稿日期` (L288)           | 页脚渲染用                                               |
| `参考编号` / `DOI` (L289) | 页脚渲染用                                               |

**现状并非错误**——作者字段 `林清河,金麦烹饪学府·副教授` 的逗号后面包含了学院和职称信息，只是渲染时全部塞在同一行，无法拆分到独立 UI 位上。

**建议：** 不改提示词。当前的紧凑格式更适合 token 预算有限的场景，少几个 footer 元素不影响效果。仅列出供参考。

## 三、🟡 中度：`小剧场格式.txt` 总控文件的问题

### 3.1 指令语言不一致

总控文件几乎全英文，但 6 个子类型文件的 `[Task]`/`[Requirement]`/`[Format]` 行也是英文，而字段定义和 Example 全部是中文。考虑到你的用户主要使用中文 AI 模型以及中文叙事环境，这种中英混杂可能导致：

- AI 在输出时混入英文（尤其是 tag name/key name 层面的英文化倾向）
- 增加不必要的上下文切换开销

**建议：** 6 个子文件的 `[Task]`、`[Requirement]`、`[Format]` 行统一改为中文。总控文件可以保持英文（因为它更像系统级指令，且内容较短）。或者反过来，全部统一到一种语言。

示例（以美食点评为例）：

```diff
-[Task: Generate a "Food Review / Ranking"]
-[Requirement: Content must be closely related to the current conversation location, and comply with the "Honest Taste" principle]
-[Format: Structured text within <food_review> tags as defined below. DO NOT output HTML/CSS/JS.]
+[任务：生成"美食点评/排行榜"]
+[要求：内容必须与当前对话场景的地点密切相关,并遵循"诚实之味"原则]
+[格式：在 <food_review> 标签内输出结构化文本,不要输出 HTML/CSS/JS]
```

### 3.2 总控 §2 Tag Format Rules 与实际存在矛盾

总控 L12 写：

> Inside the outer tag, use **a small number of sub-tags** to mark content blocks

但实际上各子类型的 sub-tag 数量差异很大——美食点评有 2 个 (`<restaurant>`, `<review>`)，学术论文有 2 个 (`<academic_paper>`, `<term_note>`)，社交动态有 2 个，而烹饪游戏和民间传说只有 1 个。"a small number" 这个说法模糊但不误导，可以保留。

### 3.3 总控 §3 Content Principles 与子文件有冗余

总控 L18-22 列出的内容原则（世界观一致、诚实之味、味阶差异、世界历法、世界术语）在每个子文件的 `[Requirement]` 和 `[Rules]` 中都被重复提及。这不算错误，但会浪费 token。

**建议：** 两种策略选一：

1. **信任总控**：从子文件的 `[Rules]` 中删除与总控 §3 重叠的通用规则，只保留该类型独有的规则
2. **信任子文件**：从总控 §3 中删除通用原则，让子文件各自声明

策略 1 节省更多 token（总控只被发送一次，子文件只激活一个）。

## 四、🟢 轻度：Example 质量与世界观一致性

### 4.1 美食点评 Example — 营业时间使用了现实制

Example L57 写 `营业：午市 11:00-14:00 | 晚市 17:30-21:00`，使用的是现实世界的 24 小时制。但世界观在 `历法与时间体系.txt` 中定义了六个「味时」（晨鲜时 04:00-08:00 等），总控 L21 也要求 "Use the world's calendar system for all dates"。

**建议：** Example 中的营业时间改用世界历法的味时表达：

```diff
-营业：午市 11:00-14:00 | 晚市 17:30-21:00
+营业时间：午浓时 - 暮醇时 | 夜甘时前半段
```

或保留现实时间但标注味时名。核心是 Example 要示范世界历法的使用，不然 AI 输出会倾向于用 24h 制。

### 4.2 社交动态 Example — 文风与角色不够多样

当前唯一的 Example 是一个厨师发帖（`铁锅小皇帝`）。`[Rules]` L30 要求 "不同类型的帖子语气差异明显：厨师分享专业自豪，食客动态轻松随意，吐槽类幽默自嘲"，但只有一个 Example 无法充分示范这些差异。

**建议（可选）：** 考虑增加一个极简的食客吐槽型 Example（2-3行正文即可），让 AI 看到语气差异的参照。但如果 token 紧张，当前的单 Example 也够用。

### 4.3 美食点评 Example — 女性主导社会特征未体现

世界观 `世界核心准则_-_诚实之味.txt` L17-20 强调女性主导社会：点评家仅限女性，女性评价具有裁定权。但美食点评的 Example 中，点评家 `「银匙」` 没有明确性别标识，两个食客也是中性的。`文风.txt` 的 `female_dominant_society` 部分要求"女性为默认性别"。

**建议：** 在 Example 中至少让点评家角色有女性倾向的表达（因为点评家必须是女性），或在 `[Rules]` 中加一句提醒：

```diff
+- 点评家角色必须为女性（世界观设定）
```

### 4.4 新闻通讯 — Example 较长，可能鼓励 AI 冗长输出

新闻通讯提供了两个完整 Example（报纸新闻 + GMA通告），总计 ~35 行。这是所有子类型中最长的 Example 部分。较长的 Example 可能引导 AI 输出更长的内容，但新闻通讯本身确实需要示范两种不同的文体，所以可以接受。

## 五、🟢 轻度：提示词结构与精简建议

### 5.1 所有子文件的 `Output format：` 前面有一个空行

6 个子文件中，`Output format：` 前面都有一个空行（如美食点评 L6-7）。这不影响功能，但统一删掉可以节省微量 token。

### 5.2 烹饪游戏菜品格式过于复杂

烹饪游戏的菜品格式 `菜名=emoji|食材1+食材2|烹饪方法|评级|点评` 用 `=` 和 `|` 和 `+` 三种分隔符嵌套，加上中文分号分隔各道菜。这个密度对 AI 来说是可以处理的，但偶尔会出现格式错误（比如 AI 在点评中使用了 `|`，导致解析错位）。

**建议（可选）：** 在 `[Rules]` 中加一条：

```diff
+- 菜品的「点评」中不要使用 | 符号,以免干扰解析
```

### 5.3 美食点评的 `[Rules]` 中 `<!-- 注释 -->` 用法

L31-32 使用 HTML 注释 `<!-- -->` 来标注数量规范。这对 AI 来说是可以理解的，但混用 XML 标签和 HTML 注释可能在某些模型上造成困惑。建议统一使用 `[Rules]` 的普通文本行：

```diff
-<!-- 每家餐厅后紧跟该店的1-2条review -->
-<!-- 共输出1-3家餐厅 -->
+[注意：每家餐厅后紧跟该店的1-2条review,共输出1-3家餐厅]
```

同样的改法适用于学术论文 L22 和社交动态 L26。

### 总体评价

这套提示词的设计**整体质量很高**——每个子类型都有清晰的结构定义、合理的 Rules 约束和高质量的 Example。XML 标签 + 键值对的混合格式在 AI 解析和前端渲染之间找到了很好的平衡。**主要问题集中在提示词与渲染脚本之间的字段名同步**（第一部分），这是最需要优先修复的。
